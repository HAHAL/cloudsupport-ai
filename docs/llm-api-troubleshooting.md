# LLM API 错误码排查指南

本文档用于大模型 API 售后支持场景，帮助一线支持工程师将 API 报错转换为可执行的排查路径。

## 401 Unauthorized

### 现象

- 返回 `401 Unauthorized`。
- 错误信息包含 `invalid api key`、`unauthorized`、`token expired`、`signature error`。
- 本地测试成功，但服务器或容器环境失败。

### 可能原因

- API Key 配置错误、过期、被禁用或没有正确加载。
- Authorization Header 缺失或格式错误。
- 签名算法、时间戳或 AK/SK 不匹配。
- 请求使用了错误的 Base URL、项目、工作空间或模型端点。

### 排查步骤

1. 确认运行环境中的 API Key 是否存在，避免只在本地 shell 中配置。
2. 使用最小请求验证 Key 是否可用。
3. 检查 Base URL、模型名称、工作空间和 SDK 版本。
4. 检查 Authorization Header、签名字段和请求时间是否正确。
5. 对比本地、服务器、容器中的环境变量差异。

### 需要客户补充的信息

- 完整错误响应体。
- request ID / trace ID。
- 请求时间与时区。
- Base URL、模型名、SDK 版本。
- 脱敏后的请求 Header 和鉴权配置方式。

### 临时解决方案

- 重新生成或轮换 API Key。
- 使用最小请求确认鉴权链路。
- 检查部署环境变量加载方式并重启服务。

### 是否需要升级研发

如果 Key、权限、Base URL、时间戳和签名均确认正确，但仍稳定返回 401，需要升级平台侧查询 request ID。

## 403 Forbidden

### 现象

- 返回 `403 Forbidden`。
- 错误信息包含 `permission denied`、`model access denied`、`region not allowed`。

### 可能原因

- 当前账号没有目标模型、项目或工作空间权限。
- 模型没有开通，或不支持当前区域。
- IP 白名单、安全策略、组织策略限制访问。
- 账号角色权限不足。

### 排查步骤

1. 确认账号、项目和工作空间是否匹配。
2. 检查模型是否已开通，是否支持当前区域。
3. 检查 IAM/RAM 权限、角色和策略。
4. 检查 IP 白名单、安全组、企业代理或访问控制策略。
5. 用同一账号调用其他已知可用模型做对比。

### 需要客户补充的信息

- 账号或工作空间标识。
- 模型名称和区域。
- request ID。
- 错误响应体。
- 权限策略截图或配置说明。

### 临时解决方案

- 切换到已授权模型或区域。
- 使用有权限的账号验证请求。
- 临时放通必要的网络或 IP 策略。

### 是否需要升级研发

如果控制台显示权限已开通，但 API 持续返回 403，需要升级权限或网关侧排查。

## 400 Bad Request

### 现象

- 返回 `400 Bad Request`。
- 错误信息包含 `invalid parameter`、`model not found`、`invalid message format`、`context_length_exceeded`。

### 可能原因

- 模型名称错误。
- 请求 JSON 格式错误。
- messages 结构不符合 API 要求。
- temperature、max_tokens、tools 等参数类型或范围不合法。
- 上下文长度超过模型窗口。

### 排查步骤

1. 使用最小请求体复现。
2. 校验 JSON 格式和 messages 数组结构。
3. 核对模型名称、API 版本和 endpoint path。
4. 统计输入 token 与输出 token 配置。
5. 移除非必要参数，逐步恢复定位问题字段。

### 需要客户补充的信息

- 脱敏后的完整请求体。
- 模型名称。
- SDK 版本。
- 错误响应体。
- token 统计或上下文长度估算。

### 临时解决方案

- 缩短 prompt 和 RAG context。
- 使用正确模型名。
- 降低 max_tokens。
- 移除不确定参数后重试。

### 是否需要升级研发

若请求体符合文档且最小请求仍返回 400，需要提供 request ID 升级排查。

## 429 Too Many Requests

### 现象

- 返回 `429 Too Many Requests`。
- 错误信息包含 `rate limit exceeded`、`quota exceeded`、`RPM`、`TPM`、`QPS`。

### 可能原因

- 请求频率超过 QPS/RPM。
- 输入或输出 token 超过 TPM。
- 并发数过高。
- 账号额度、账单或模型配额不足。
- 重试策略过于激进，形成放大流量。

### 排查步骤

1. 查看请求峰值、并发数、RPM、TPM。
2. 检查是否有自动重试或循环调用。
3. 查看账号额度、账单状态和模型配额。
4. 检查是否多个服务共享同一个 API Key。
5. 为 429 增加指数退避和 jitter。

### 需要客户补充的信息

- 请求时间范围。
- 平均 QPS、峰值 QPS、并发数。
- 输入 token、输出 token、模型名。
- 是否启用重试。
- 错误响应体和 request ID。

### 临时解决方案

- 降低并发。
- 增加指数退避。
- 减少 prompt 和输出长度。
- 拆分流量或申请更高配额。

### 是否需要升级研发

如果配额显示充足但仍持续 429，需要提供 request ID、时间范围和调用量数据升级排查。

## 408 / 504 / timeout

### 现象

- 客户端超时。
- 网关返回 408 或 504。
- 流式接口长时间无输出。

### 可能原因

- 请求体过大，上下文过长。
- 模型推理耗时超过客户端或网关超时。
- 网络链路不稳定。
- 代理、Nginx 或负载均衡超时配置过短。
- 下游依赖响应慢。

### 排查步骤

1. 记录端到端耗时、TTFB、首 token 时间和总输出时间。
2. 降低输入长度和 max_tokens 后重试。
3. 对比流式和非流式调用。
4. 检查客户端、Nginx、网关和负载均衡超时配置。
5. 检查是否跨地域访问或经过企业代理。

### 需要客户补充的信息

- request ID。
- 请求时间和时区。
- 请求体大小、输入 token、输出 token。
- 客户端 timeout 配置。
- 是否使用代理或网关。

### 临时解决方案

- 缩短上下文。
- 增加客户端超时。
- 使用流式输出。
- 对超时错误增加幂等安全的重试。

### 是否需要升级研发

如果短请求也稳定超时，或同一区域大量请求超时，需要升级平台侧排查。

## 5xx

### 现象

- 返回 `500`、`502`、`503`、`504`。
- 偶发失败，重试后成功。
- 多个模型或 endpoint 同时异常。

### 可能原因

- 模型服务临时异常。
- 上游服务不可用。
- 网关、代理或负载均衡异常。
- 请求触发未预期服务端错误。

### 排查步骤

1. 记录 request ID、模型名、时间、区域和请求体大小。
2. 检查是否可通过重试恢复。
3. 对比不同模型、不同区域、不同 endpoint。
4. 检查客户端代理和网络链路。
5. 对 5xx 使用指数退避和熔断。

### 需要客户补充的信息

- request ID。
- 错误响应体。
- 模型名和 endpoint。
- 请求时间和时区。
- 是否批量出现。

### 临时解决方案

- 增加指数退避重试。
- 使用备用模型或备用 endpoint。
- 对关键业务增加降级策略。

### 是否需要升级研发

持续 5xx、同区域大面积异常、或 request ID 可稳定复现时需要升级。

## stream interrupted

### 现象

- 流式输出中途断开。
- 客户端收到半截内容。
- 代理或网关报连接重置。

### 可能原因

- 客户端 read timeout 过短。
- Nginx、网关或代理缓冲/超时配置不适配流式响应。
- 网络中断。
- 输出过长导致连接持续时间过久。

### 排查步骤

1. 对比流式和非流式请求。
2. 检查客户端 read timeout。
3. 检查 Nginx、网关和代理的 streaming 配置。
4. 降低 max_tokens 后复测。
5. 记录断开时间点和已输出 token 数。

### 需要客户补充的信息

- request ID。
- 客户端 SDK 版本。
- 代理或网关配置。
- 断开前输出内容长度。
- 是否稳定复现。

### 临时解决方案

- 增加 read timeout。
- 关闭代理缓冲。
- 减少单次输出长度。
- 增加客户端断线重试和续写策略。

### 是否需要升级研发

如果排除客户端和代理配置后仍稳定中断，需要升级。

## context_length_exceeded

### 现象

- 返回 `context_length_exceeded`。
- 模型提示输入超过上下文窗口。
- RAG 问答在拼接多个文档后失败。

### 可能原因

- 用户输入过长。
- RAG Top-K 过大。
- chunk 太大。
- 历史对话未裁剪。
- max_tokens 设置过高。

### 排查步骤

1. 统计 prompt、history、RAG context 和 max_tokens。
2. 减少 Top-K。
3. 降低 chunk_size 或做摘要压缩。
4. 裁剪历史对话。
5. 使用支持更长上下文的模型。

### 需要客户补充的信息

- 请求体。
- 模型名。
- prompt token 估算。
- RAG Top-K 和 chunk_size。
- 是否包含历史对话。

### 临时解决方案

- 裁剪上下文。
- 减少引用文档数量。
- 对长文档先摘要再问答。
- 使用长上下文模型。

### 是否需要升级研发

通常不需要升级；若 token 统计明显低于模型限制但仍报错，再升级排查。
